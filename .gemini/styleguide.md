# Bookscape 프로젝트 코드 스타일 가이드

## 기본 지침

- 모든 리뷰 코멘트, 요약, 제안은 반드시 **한국어(Korean)**로 작성하세요.
- **충분한 근거 없이 단정하지 마세요.** 확실하지 않은 경우 "알 수 없음/확실하지 않음"을 명시하고 추측일 경우 추측임을 밝히세요.
- 가능하면 공식 문서 등 **출처/근거**(공식 문서)를 첨부하고, 답변은 **ES6+** 기준을 따릅니다.
- 답변은 전문적이고 친절한 시니어 개발자의 어조를 유지하세요.
- 아래 명시된 최신 버전을 기준으로 답변해야 하며, 이전 버전의 문법이나 기능을 제안하지 마세요.
  - **Framework**: Next.js 16.2.4 (App Router)
  - **Library**: React 19.2.4
  - **Language**: TypeScript ^5
  - **Styling**: Tailwind CSS v4

## 아키텍처 및 컴포넌트 규칙

- **Lite FSD 구조 준수**:
  - `app/`: Next.js 라우팅 및 레이아웃.
  - `features/`: 도메인별 비즈니스 로직 및 UI.
  - `shared/`: 공용 UI, API, 유틸리티.
- **의존성 규칙**: 하위 계층(`shared`)이 상위 계층(`features`, `app`)을 참조(import)하는지 엄격히 확인하세요.
- **단일 책임 원칙 (SRP)**: 함수와 컴포넌트는 하나의 책임만 가집니다. 비대해질 경우 작은 컴포넌트로 분해하거나 UI/로직을 분리하세요.
- **패턴 공통화**: 반복되는 코드 패턴은 UI 컴포넌트로 승격하여 재사용합니다.

## 네이밍 규칙

- **디렉토리**: `kebab-case`
- **파일명**:
  - 컴포넌트/페이지: `PascalCase.tsx`
  - 훅, 유틸, API, 일반 모듈: `camelCase.ts`
  - Assets(아이콘, 이미지): `kebab-case`
- **함수/변수**:
  - 페이지 함수: `PascalCase` + `Page` 접미사 필수 (예: `MainPage`).
  - 컴포넌트 함수: `PascalCase`
  - 일반 함수/변수명: `camelCase`
  - 상수: 기본 상수는 `UPPER_SNAKE_CASE` (예: `API_BASE_URL`). **단, cva 같은 스타일 상수**는 `camelCase`를 사용합니다.
- **이벤트 핸들러**: 선언 시 `handle + 동사`, Prop으로 전달 시 `on + 동사` (예: `handleSubmit`, `onSubmit`)
- **Boolean 변수**: 질문 형태의 접두사(`is`, `has`) 필수 (예: `isLoading`, `hasToken`)

## 에셋 및 스타일링

- **에셋 관리**:
  - 아이콘/로고/일러스트는 `SVG`, 투명 이미지나 UI 요소는 `PNG`를 사용하며 대용량은 경량화합니다.
  - 넘버링이 필요할 시 2자리 숫자로 통일합니다 (예: `01`, `02`).
  - 에셋 폴더 내 `index.ts`를 활용하여 `import { ArrowIcon } from index.js` 형태로 내보냅니다.
- **Tailwind CSS 전용**: 스타일링은 기본적으로 Tailwind를 사용합니다. 인라인 스타일이나 다른 CSS-in-JS 혼용을 지양하세요.
- **시맨틱 태그 준수**: `div`, `span` 남발을 피하고 의미 있는 태그(`<article>`, `<section>`, `<nav>`, `<footer>` 등)와 헤딩 계층을 준수하는지 확인하세요.
- **웹 접근성(A11y)**:
  - 의미 있는 `alt` 속성 필수 (장식용은 빈 값 `alt=""`).
  - 클릭 요소에는 올바른 태그(`<button>`, `<a>`)를 사용하고 있는지 확인하세요.
  - ARIA 레이블은 필요한 곳에만 적절히 사용합니다.

## TypeScript 및 문법 규칙

- **Type 선언**: `interface` 사용을 금지하고, 오직 **`type`**만 사용합니다.
- **절대 경로**: 상대 경로(`../../`) 대신 **절대 경로 별칭(`@/`)**을 사용하고 있는지 확인하세요.
- **가독성**: 과도한 중첩(>2-3단계)을 피하고 가독성 높은 코드를 지향합니다.
- **함수 선언 방식**:
  - 페이지 및 컴포넌트: **선언식** 사용 (`export default function Page() {}`)
  - 그 외 유틸, 훅 등 모든 함수: **화살표 함수** 사용 (`const change = () => {}`)

## 주석 가이드

- **TSDoc**: 컴포넌트 상단에 `/** ... */` 형태의 설명을 작성합니다.
  - 내부에 가급적 `@example` 코드 스니펫을 포함하도록 권장해 주세요.
  - 타입스크립트를 사용하므로 **`@params` 태그는 절대 사용하지 않습니다.**
- **특수 주석 태그 활용**: 상황에 맞게 아래의 접두사를 사용했는지 확인하고 권장하세요.
  - `// TODO:` : 이후에 처리해야 할 미구현 작업
  - `// BUG:` : 해결해야 할 버그가 있는 경우
  - `// NOTE:` : 제약 조건, 중요한 동작 방식 등 모르면 안 되는 필수 정보
  - `// OPTIMIZE:` : 리팩토링이나 성능 개선(useMemo 등)이 필요한 경우
  - `// INFO:` : 백엔드 의존성 등 알아두면 좋은 참고용 정보
